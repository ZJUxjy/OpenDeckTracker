## 5. `add-hearthmirror-bridge`（依赖 spike 01 + 02 通过）

**类型**：业务能力 change，建立 `packages/hearthmirror/` 正式实现。

**范围**：

- `packages/hearthmirror/native/`：64 位 Rust crate，napi-rs 暴露 Mono 解析 API。
- 必须遵循 ADR 0001 的 Engineering Constraints：`RemotePtr(u32)` newtype、`#[napi] fn -> Result<T>`、`clippy::unwrap_used` CI 门禁。
- 实现 `IReflection` 的核心 12 方法：`GetCollection` / `GetMatchInfo` / `GetMedalInfo` / `GetDecks` / `GetBattleTag` / `GetAccountId` / `GetGameType` / `IsSpectating` / `IsGameOver` / `GetArenaState` / `GetArenaDeck` / `GetServerInfo`。
- `packages/hearthmirror/`：TypeScript 高层 API（`HearthMirror` 类、类型定义、错误处理）。
- `apps/desktop/src/main/ipc.ts`：`hearthmirror:*` 通道，渲染进程通过 `window.hdt.hearthmirror.*` 调用。

## 6. `add-deck-management`

**类型**：业务能力 change。依赖：`add-card-database`（已完成）。

**范围**：

- 新建 `packages/core/deck` + 引入 `better-sqlite3` 与 `zustand`。
- 卡组 CRUD（数据库 schema + migration）。
- 卡组合法性校验（30 张、同名 ≤ 2、传说 ≤ 1、职业限制）。
- 卡组导入（剪贴板文本、卡组码、JSON）/ 导出（卡组码、JSON）。
- Renderer 卡组选择器与编辑器（基于现有 `Dashboard.tsx` 与 `Decklist.tsx` 扩展）。

## 7. `add-hearthwatcher`

**类型**：业务能力 change。

**范围**：监听 Hearthstone Logs 目录，解析 PowerTaskList / LoadingScreen / Asset / Rank 等日志，构建实时 GameState。涉及的文件量大，建议先做 PowerTaskList 解析器与 GameState 模型，其余日志类型留给单独 change。

## 8. `add-overlay-window`

**类型**：业务能力 change。依赖：`add-hearthwatcher` 与 `add-deck-management`。

**范围**：独立透明 BrowserWindow + Hearthstone 窗口位置追踪 + 己方/对手卡组面板 + 回合计时器。OverlayView 从 `apps/desktop/renderer` 拆出到独立 BrowserWindow（当前 `/overlay` 路由仅做主窗口内的演示）。
