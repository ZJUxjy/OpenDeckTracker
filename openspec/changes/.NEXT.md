## hearthmirror-rs 集成路线图（Plan ↦ 3 changes 拆分）

> 来源：[`docs/superpowers/plans/2026-04-20-integrate-hearthmirror-rs.md`](../../docs/superpowers/plans/2026-04-20-integrate-hearthmirror-rs.md) §1-§8。
> 该 plan 已拆为下述 3 个 OpenSpec changes，按依赖顺序执行：

### ✓✓ 5d. `verify-hearthmirror-on-real-hs`（spike，已实施）

**类型**：spike / 数据收集。依赖：`add-hearthmirror-reflection-methods`（已完成）。

**结果**：✅ 高价值产出。1-2h 投入，抓到 4 个对后续工作有方向性影响的事实：

- **F-1 (Critical, P0 blocking)**：`MonoRuntime::init()` 在真实炉石上以 `STATUS_ACCESS_VIOLATION` 崩溃。Root cause 已定位到 `runtime.rs:97` 的 `mono.size.min(0x100_000)` 1MB PE 读取上限把 `pelite::PeView::module` 喂坏 — `exports.by().name()` 跟随 RVA 越界读触发 Windows SEH。**12 个反射方法全部 blocked**。修复见 5d-fix。
- **F-2 (Positive)**：去 cap 后 init 链 6 步全绿（process / handle / module / PE / disasm pattern A / root domain `0x0B442E70`）— 现有架构无更深层错误。
- **F-3 (Informational)**：mono dll 实际位置 `MonoBleedingEdge/EmbedRuntime/mono-2.0-bdwgc.dll`，非 `Rewrite_Design.md` 假设的 `Mono/`。
- **F-4 (Informational, ⚠️ critical implication for 5e)**：**生产炉石 Unity 版本 = 2022.3.62f2**，不是计划中的 2021.3。

参考：[`openspec/changes/verify-hearthmirror-on-real-hs/`](verify-hearthmirror-on-real-hs/) + [`docs/spikes/0003-*.md`](../../docs/spikes/0003-hearthmirror-reflection-runtime-validation.md)

### ✓ 5d-fix. `fix-hearthmirror-pe-read-cap`（已提案 — P0 阻塞修复）

**类型**：bugfix。依赖：无。**5e/5f 的硬阻塞** — 不修，所有真机验证全部失败。

**范围**：把 `runtime.rs:97` 的 `let pe_size = mono.size.min(0x100_000) as usize;` 改为 `let pe_size = mono.size as usize;`（一行 fix），加 1 个回归集成测试，重跑 spike 0003 Run 2 验证 12 个反射方法的实际状态。源于 spike 0003 R-1。

**预估**：30 分钟（1 行代码 + 1 个测试 + 重跑 spike）。

参考：[`openspec/changes/fix-hearthmirror-pe-read-cap/`](fix-hearthmirror-pe-read-cap/)

### ✓ 5e. `add-hearthmirror-offset-probing`（已提案 — 等 5d-fix 后再启动）

**类型**：稳定性 / 基础设施。依赖：`fix-hearthmirror-pe-read-cap`（强阻塞）+ `verify-hearthmirror-on-real-hs` Run 2 数据。

> ⚠️ **设计需修订**：原 spec 写的 baseline 是 `unity-2021.3.json`，但 spike 0003 F-4 揭示生产是 **Unity 2022.3.62f2**。启动 5e 前必须：(a) 把 baseline 改名/换源为 `unity-2022.3.json`，或 (b) 从 hearthmirror-rs 拿对应版本，或 (c) 用 5d-fix 后的 Run 2 实测数据反推 baseline。

**范围**：移植 hearthmirror-rs 的 `iced-x86` 反汇编引擎 + `MonoOffsets` JSON 配置 + `OffsetProber`（6 critical + 4 best-effort probes），替换 `field_paths.rs` 中硬编码的 13 个 Mono 结构偏移与 `runtime.rs` 中两个手写 byte-pattern 探测。业务字段名常量与 12 个反射方法链路保持不变。兑现 ADR 0001 约束 #5。

**预估**：4-6h（含 baseline 修订）。

参考：[`openspec/changes/add-hearthmirror-offset-probing/`](add-hearthmirror-offset-probing/)

### ✓ 5f. `add-hearthmirror-image-walking`（已提案）

**类型**：稳定性 / 内部重构。依赖：`add-hearthmirror-offset-probing`（必须先完成 — 用 `image.class_cache` / `class.parent` 等偏移）。

**范围**：移植 hearthmirror-rs 的 `MonoImage` (class_cache hashtable walking) + `MonoFieldDef` 独立模块 + `MonoClass.fields_recursive`（含继承链）。`MonoRuntime::find_class` 内部从 token 反向探测重构为 hashtable walking；删除 `probe_class_def_table_offset`。napi 公共 API 与 12 个反射方法链路不变。

参考：[`openspec/changes/add-hearthmirror-image-walking/`](add-hearthmirror-image-walking/)
